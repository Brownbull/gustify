# Story 4.2: Exploration Profile & Proficiency Tracking

## Intent
**Epic Handle:** "Cook it, log it, grow from it"
**Story Handle:** "This story shows you the stamps in your cooking passport -- where you've been and how far you've come."

## Status: drafted

## User Story
As a user, I want to see my culinary growth -- cuisines tried, techniques learned, proficiency tier -- so that I can feel my progress.

## Acceptance Criteria

### AC-1: Profile Update on Meal Log
- **Given** a meal is logged via "Lo cocine!"
- **When** the cooking log saves
- **Then** my `cookingProfile` is updated: new cuisines added to `cookedCuisines`, new techniques to `cookedTechniques`, new ingredient canonicalIds to `cookedIngredients`

### AC-2: Proficiency Tier Calculation
- **Given** my cooking history
- **When** average complexity is computed from all cooked meals
- **Then** my proficiency tier is derived: Principiante (1.0-2.0), Comodo (2.0-3.0), Aventurero (3.0-4.0), Avanzado (4.0-5.0)

### AC-3: Profile Display
- **Given** the More page
- **When** I view my exploration profile section
- **Then** I see: proficiency tier badge, cuisines tried (count + chip list), techniques used (count + chip list), total dishes cooked, average complexity

### AC-4: Zero State
- **Given** no meals have been cooked
- **When** viewing the exploration profile
- **Then** it shows "Principiante" tier with "0 platos cocinados" and encouragement to cook the first meal

## Architectural ACs

- **AC-ARCH-LOC-1:** ExplorationProfile component at `src/components/ExplorationProfile.tsx`
- **AC-ARCH-PATTERN-1:** Profile update happens in the cooking log service (same transaction as meal log if possible)
- **AC-ARCH-PATTERN-2:** cookingProfile already exists in Firestore user document -- this story updates it, not creates it
- **AC-ARCH-NO-1:** Proficiency tier is never set manually -- always derived from cooking history

## Dev Notes

- The `CookingProfile` type already has all needed fields (`cookedCuisines`, `cookedTechniques`, `cookedIngredients`, `avgComplexity`, `proficiencyTier`, `dishesCooked`). No type changes needed.
- The profile update should use Firestore `arrayUnion` for adding to cuisines/techniques/ingredients sets -- this handles deduplication atomically.
- Computing `avgComplexity` requires either: (a) reading all cooked meals and averaging, or (b) maintaining a running average with `dishesCooked` counter. Option (b) is more efficient: `newAvg = (oldAvg * oldCount + newComplexity) / (oldCount + 1)`.
- The proficiency tier is implicit -- users never select it. It's always derived. This is a core design principle.
- The ExplorationProfile component is read-only. It renders data from the user's cookingProfile.
