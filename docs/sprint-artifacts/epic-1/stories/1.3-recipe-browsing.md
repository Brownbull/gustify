# Story 1.3: Recipe Browsing & Pantry Match Ranking

## Intent
**Epic Handle:** "Your pantry-ranked recipe library"
**Story Handle:** "This story opens the recipe library doors and ranks every book by what's in your fridge."

## Status: drafted

## User Story
As a user, I want to browse all recipes ranked by how many ingredients I already have, so that I can find meals I can cook right now.

## Acceptance Criteria

### AC-1: Recipe List Sorted by Pantry Match
- **Given** a user with pantry items
- **When** they open the Recipes tab
- **Then** they see all recipes from the pool sorted by pantry match % (highest first)

### AC-2: Pantry Match Calculation
- **Given** a recipe with 8 ingredients
- **When** 6 match the user's pantry (by canonicalId or normalized name)
- **Then** pantry match shows 75%

### AC-3: Loading State
- **Given** recipes are being fetched
- **When** the Recipes tab is open
- **Then** a loading spinner/skeleton is shown

### AC-4: Empty State
- **Given** no recipes exist in the pool
- **When** the user opens the Recipes tab
- **Then** an empty state message is shown

### AC-5: Error State
- **Given** the Firestore query fails
- **When** the user opens the Recipes tab
- **Then** an error message with retry option is shown

## Architectural ACs

- **AC-ARCH-LOC-1:** RecipesPage at `src/pages/RecipesPage.tsx`
- **AC-ARCH-PATTERN-1:** RecipeStore subscribes to `recipes/` collection via real-time listener
- **AC-ARCH-PATTERN-2:** Pantry match computed client-side by comparing recipe ingredients against pantryStore items
- **AC-ARCH-NO-1:** No Firestore SDK imports in RecipesPage — all data flows through stores
- **AC-ARCH-PATTERN-3:** Recipe list wrapped in ErrorBoundary for graceful degradation

## Dev Notes

- The existing RecipesPage has a "Generate suggestions" button calling Gemini. This entire flow is removed in this story. Recipe generation moves to Epic 3 (Recipe Contribution Pipeline).
- Keep the existing `RecipeDetailModal` for now — Story 1.5 will replace it with a full page + React Router.
- The pantry match computation currently lives in `recipeStore.ts` as `computePantryMatch()`. Extract it as a pure utility function and reuse it with the new subscription-based data.
- Real-time listener on `recipes/` should have `limit(200)` to avoid unbounded reads (code-review-patterns.md: real-time query limits).
- The recipeStore should cross-read from pantryStore to get current pantry items for match calculation. This is one-way coupling (acceptable per hardening Pattern 6).
