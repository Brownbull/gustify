# Story 1.1: Recipe Data Model & Zod Validation

## Intent
**Epic Handle:** "Your pantry-ranked recipe library"
**Story Handle:** "This story builds the shelves for your pantry-ranked recipe library by defining what a recipe IS."

## Status: drafted

## User Story
As a developer, I want a Zod-validated recipe schema and Firestore service, so that all recipe data is type-safe and consistent across contribution paths.

## Acceptance Criteria

### AC-1: Zod Schema Validates Recipes
- **Given** a valid recipe JSON object
- **When** validated against RecipeSchema
- **Then** it produces a typed Recipe object with all required fields

### AC-2: Zod Schema Rejects Invalid Recipes
- **Given** a recipe JSON missing required fields (e.g., no name, no ingredients)
- **When** validated against RecipeSchema
- **Then** it returns field-level error messages identifying each missing/invalid field

### AC-3: Recipe Service Reads from Firestore
- **Given** the `recipes/` Firestore collection contains documents
- **When** `getAllRecipes()` is called
- **Then** documents are converted to typed Recipe objects with `id` from `doc.id`

### AC-4: Validation Edge Cases
- **Given** edge case inputs (empty ingredients array, complexity 0 or 6, empty steps, extremely long name)
- **When** validated against RecipeSchema
- **Then** appropriate validation errors are returned for out-of-bounds values

## Architectural ACs

- **AC-ARCH-LOC-1:** Zod schema defined in `src/types/recipe.ts` alongside existing interfaces
- **AC-ARCH-PATTERN-1:** Schema follows `{Type}Schema` naming, type inferred via `z.infer<typeof RecipeSchema>`
- **AC-ARCH-LOC-2:** Recipe Firestore service at `src/services/recipes.ts` with `docToRecipe()` converter
- **AC-ARCH-PATTERN-2:** Service uses `recipesPath()` private helper for collection path (no hardcoded strings)
- **AC-ARCH-NO-1:** No Firestore SDK imports in components or pages

## Tasks

### Task 1: Extend Recipe Types with Zod Schema (3 subtasks)
- [ ] 1.1: Add `zod` dependency to package.json
- [ ] 1.2: Define `RecipeSchema` in `src/types/recipe.ts` with fields: name (string, 1-200 chars), description (string), cuisine (string), techniques (string array, min 1), complexity (int 1-5), prepTime (number >= 0), cookTime (number >= 0), servings (int >= 1), ingredients (array of {name, quantity, unit, canonicalId?}), steps (array of {order, instruction, duration?})
- [ ] 1.3: Export `StoredRecipe` type via `z.infer<typeof RecipeSchema>` — keep existing `Recipe` interface (with `pantryMatchPct`, `noveltyBadges`) as the client-enriched version

### Task 2: Create Recipe Firestore Service (4 subtasks)
- [ ] 2.1: Create `src/services/recipes.ts` with private `recipesPath()` returning `'recipes'`
- [ ] 2.2: Implement `docToRecipe(doc)` converter following existing `docTo{Type}` pattern
- [ ] 2.3: Implement `getAllRecipes(): Promise<StoredRecipe[]>` — query `recipes/` collection, convert all docs
- [ ] 2.4: Implement `getRecipeById(recipeId: string): Promise<StoredRecipe | null>` — single doc fetch

### Task 3: Hardening — Validation Edge Case Tests (4 subtasks)
- [ ] 3.1: Test valid recipe passes schema validation
- [ ] 3.2: Test missing required fields produce field-level errors (name, ingredients, steps, cuisine)
- [ ] 3.3: Test boundary values: complexity 0 (reject), 1 (accept), 5 (accept), 6 (reject); servings 0 (reject), 1 (accept); empty ingredients array (reject); empty steps array (reject)
- [ ] 3.4: Test string constraints: name > 200 chars (reject), empty name (reject)

## Dev Notes

- The existing `Recipe` interface has `pantryMatchPct` and `noveltyBadges` which are computed client-side, not stored in Firestore. The Zod schema (`RecipeSchema`) defines the stored shape. Keep both: `StoredRecipe` for Firestore, `Recipe` for UI.
- The existing `GeminiRecipe` interface can be replaced by `RecipeSchema.omit({ id: true })` or similar — evaluate during implementation.
- Follow the exact pattern from `src/services/ingredients.ts` for service structure.
- Firestore rules for `recipes/` should allow read for all authenticated users. Write rules will be `false` (CF-only writes) — update in Story 1.2 when seeding.
