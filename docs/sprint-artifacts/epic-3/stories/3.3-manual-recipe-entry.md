# Story 3.3: Manual Recipe Entry Form

## Intent
**Epic Handle:** "Anyone can grow the recipe pool"
**Story Handle:** "This story adds a handwritten recipe card option to the community recipe box."

## Status: drafted

## User Story
As a user, I want to enter a recipe manually through a step-by-step form, so that I can contribute recipes without using AI.

## Acceptance Criteria

### AC-1: Multi-Step Form
- **Given** the "Manual" tab on the contribution page
- **When** I tap "Ingresar manualmente"
- **Then** a form appears with sections for: basic info, ingredients, steps

### AC-2: Basic Info Entry
- **Given** the form
- **When** I enter name, description, cuisine (dropdown), complexity (1-5 slider), prep time, cook time, servings
- **Then** the values are captured for the recipe

### AC-3: Ingredient Entry
- **Given** the ingredients section
- **When** I add ingredients one by one (name, quantity, unit)
- **Then** each ingredient is added to the list with option to remove

### AC-4: Step Entry
- **Given** the steps section
- **When** I add cooking steps in order
- **Then** steps are numbered automatically with option to reorder or remove

### AC-5: Validation on Submit
- **Given** the completed form
- **When** I tap "Guardar receta"
- **Then** Zod validation runs; if valid, recipe is submitted via contributeRecipe CF

### AC-6: Validation Errors
- **Given** missing required fields
- **When** I try to submit
- **Then** validation errors highlight the specific missing fields in Spanish

## Architectural ACs

- **AC-ARCH-LOC-1:** ManualRecipeForm at `src/components/ManualRecipeForm.tsx`
- **AC-ARCH-PATTERN-1:** Form state managed locally (useState/useReducer) — not in Zustand store
- **AC-ARCH-PATTERN-2:** Submit calls `submitRecipe()` from `src/services/recipe-contribution.ts` (Story 3.2)
- **AC-ARCH-PATTERN-3:** Zod validation runs client-side before CF call (fast feedback)
- **AC-ARCH-NO-1:** No direct Firestore writes — all submissions go through the contributeRecipe CF

## Tasks

### Task 1: Create ManualRecipeForm Component (5 subtasks)
- [ ] 1.1: Basic info section: name (text input), description (textarea), cuisine (select from dynamic list derived from existing recipes + "Otra"), complexity (1-5 segmented control), prep time (number), cook time (number), servings (number)
- [ ] 1.2: Ingredients section: "Agregar ingrediente" button, each ingredient row has name (text), quantity (number), unit (select: g, kg, ml, L, unidad, cucharada, taza, pizca), remove button
- [ ] 1.3: Steps section: "Agregar paso" button, each step has textarea for instruction, optional duration (minutes), reorder (up/down arrows), remove button
- [ ] 1.4: Techniques section: multi-select chips from common techniques list (sofrito, horneado, salteado, hervido, fritura, asado, marinado, al vapor)
- [ ] 1.5: Form state managed with useReducer for complex nested state (ingredients array, steps array)

### Task 2: Validation & Submission (3 subtasks)
- [ ] 2.1: On submit, build recipe object from form state and validate with RecipeSchema
- [ ] 2.2: Display field-level errors inline (Spanish messages: "El nombre es obligatorio", "Agrega al menos un ingrediente")
- [ ] 2.3: On valid, call `submitRecipe(recipe, 'manual')` from recipe-contribution service

### Task 3: Integrate into ContributeRecipePage (2 subtasks)
- [ ] 3.1: Add "Manual" tab rendering ManualRecipeForm
- [ ] 3.2: On successful submission, show confirmation and option to add another or return to recipes

### Task 4: Hardening — Input Sanitization (3 subtasks)
- [ ] 4.1: Sanitize all text inputs: name (max 200 chars), description (max 500 chars), ingredient names (max 100 chars), step instructions (max 500 chars)
- [ ] 4.2: Enforce via Zod schema maxLength constraints + client-side character counters
- [ ] 4.3: Strip leading/trailing whitespace from all text fields before validation

### Task 5: Tests (3 subtasks)
- [ ] 5.1: Test form renders all sections (basic info, ingredients, steps)
- [ ] 5.2: Test adding/removing ingredients updates the list
- [ ] 5.3: Test validation errors shown for empty required fields

## Dev Notes

- This form is the most complex UI component in the MVP. Use `useReducer` for form state management — the nested arrays (ingredients, steps) make `useState` unwieldy.
- The cuisine dropdown should include common options (Chilena, Mexicana, Italiana, Peruana, Japonesa, Tailandesa, India, Francesa, Espanola, Americana, China, Coreana, Arabe, Otra) but also allow custom entry.
- Units in Spanish: "g" (gramos), "kg" (kilogramos), "ml" (mililitros), "L" (litros), "unidad" (unit), "cucharada" (tablespoon), "cdta" (teaspoon), "taza" (cup), "pizca" (pinch).
- The form does NOT try to match ingredient names to canonicalIds — that's the CF's job (or a future enhancement). Users enter free-text ingredient names.
- Mobile-first: the form should work well on 375px width. Consider collapsible sections or a wizard-style step-by-step flow rather than showing everything at once.
- The "Guardar receta" button calls the same `submitRecipe()` service from Story 3.2 with `source: 'manual'`. This means Story 3.2 must be implemented first (or simultaneously).
