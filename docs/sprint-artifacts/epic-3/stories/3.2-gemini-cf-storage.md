# Story 3.2: Built-in Gemini Generation & CF Recipe Storage

## Intent
**Epic Handle:** "Anyone can grow the recipe pool"
**Story Handle:** "This story hires a librarian for the community recipe box -- validates, deduplicates, and files every card."

## Status: drafted

## User Story
As a user, I want to generate a recipe with one tap using the built-in Gemini integration, and have it stored in the recipe pool for everyone, so that I can contribute without leaving the app.

## Acceptance Criteria

### AC-1: One-Tap Generation
- **Given** the Gemini tab on the contribution page
- **When** I tap "Generar con Gemini"
- **Then** the app calls the Cloud Function with my context and returns a generated recipe

### AC-2: Server-Side Validation & Storage
- **Given** a generated recipe from Gemini
- **When** the Cloud Function validates it
- **Then** the recipe is stored in `recipes/` collection and available to all users

### AC-3: Duplicate Detection
- **Given** a recipe with a name similar to an existing recipe AND overlapping ingredients
- **When** the Cloud Function checks for duplicates
- **Then** the duplicate is detected (Jaccard similarity >= 0.8 on canonicalIds) and the user is notified

### AC-4: Duplicate Consolidation
- **Given** a detected duplicate
- **When** consolidation runs
- **Then** the existing recipe is preserved (no new document created), user is informed

### AC-5: Gemini Failure Graceful Degradation
- **Given** the Gemini API is unavailable or returns an error
- **When** generation fails
- **Then** a clear error message is shown: "Gemini no disponible. Usa el prompt externo para generar con otra IA."

### AC-6: Contributor Anonymity
- **Given** a user contributes a recipe
- **When** it is stored in Firestore
- **Then** no contributor identity is included in the recipe document (NFR-2.9)

## Architectural ACs

- **AC-ARCH-LOC-1:** Cloud Function at `functions/src/contributeRecipe.ts`
- **AC-ARCH-LOC-2:** Client service at `src/services/recipe-contribution.ts`
- **AC-ARCH-PATTERN-1:** CF validates with Zod schema, dedup checks, then writes via Admin SDK
- **AC-ARCH-PATTERN-2:** Client calls CF via `httpsCallable` wrapper in services/ (no direct fetch)
- **AC-ARCH-PATTERN-3:** Dedup uses normalized name comparison + Jaccard similarity on ingredient canonicalIds
- **AC-ARCH-NO-1:** No direct client writes to `recipes/` — CF mediates all writes
- **AC-ARCH-NO-2:** No contributor userId stored in recipe document

## Tasks

### Task 1: Create contributeRecipe Cloud Function (5 subtasks)
- [ ] 1.1: Create callable CF accepting recipe data + userId (for rate limiting, not stored in recipe)
- [ ] 1.2: Validate incoming recipe with RecipeSchema (Zod) — reject with field-level errors if invalid
- [ ] 1.3: Run dedup check against existing recipes (see Task 2)
- [ ] 1.4: If no duplicate: write recipe to `recipes/` with `createdAt` timestamp, `source: 'gemini' | 'external' | 'manual'`
- [ ] 1.5: Return `ServiceResult<{recipeId, isDuplicate, existingRecipeId?}>`

### Task 2: Implement Dedup Algorithm (4 subtasks)
- [ ] 2.1: Create `functions/src/lib/recipe-dedup.ts` with `findDuplicate(newRecipe, existingRecipes)`
- [ ] 2.2: Normalize recipe name: lowercase, remove accents, strip common words ("receta de", "mi")
- [ ] 2.3: Compute Jaccard similarity on ingredient canonicalId sets between new and each existing recipe
- [ ] 2.4: Duplicate if: normalized name match >= 0.85 OR (name similarity >= 0.6 AND ingredient Jaccard >= 0.8)

### Task 3: Create Client Contribution Service (3 subtasks)
- [ ] 3.1: Create `src/services/recipe-contribution.ts` with `submitRecipe(recipe, source)` using `httpsCallable`
- [ ] 3.2: Handle success (recipe stored), duplicate (show existing), and error (show message) responses
- [ ] 3.3: Expose `generateWithGemini(context)` that calls the existing `suggestRecipes` CF with `mode: 'contribute'` flag, then pipes result to `submitRecipe`

### Task 4: Integrate Gemini Tab in ContributeRecipePage (3 subtasks)
- [ ] 4.1: Add "Gemini" tab showing context summary + "Generar con Gemini" button
- [ ] 4.2: On success: show RecipePreview with "Guardado" confirmation
- [ ] 4.3: On duplicate: show existing recipe comparison with "Ya existe una receta similar" message

### Task 5: Hardening — Error Resilience (3 subtasks)
- [ ] 5.1: Handle CF timeout (30s default) with clear message and retry option
- [ ] 5.2: Handle Gemini rate limit (429) with backoff message
- [ ] 5.3: If Gemini fails, show fallback: "Usa el prompt externo" with link to external tab

### Task 6: Hardening — Cross-Store & Tests (3 subtasks)
- [ ] 6.1: Document one-way coupling: ContributeRecipePage reads pantryStore + authStore for context
- [ ] 6.2: Unit test dedup algorithm: exact name match, similar name + high ingredient overlap, different recipes pass
- [ ] 6.3: Unit test CF validation: valid recipe passes, missing fields rejected with errors

## Dev Notes

- The existing `suggestRecipes` CF generates recipes but doesn't store them. This story creates a new `contributeRecipe` CF specifically for validation + dedup + storage. The Gemini generation can reuse the existing CF or be integrated into the new one.
- **Dedup strategy (from architecture doc):** Normalize name + Jaccard similarity on ingredient canonicalIds, threshold 0.8. This catches "Arroz con Pollo" vs "Arroz Con Pollo" and recipes with near-identical ingredient lists but different names.
- The CF validates server-side even though the client also validates with Zod. Double validation is intentional (security layer).
- `source` field on stored recipes: `'seed'` (from seed script), `'gemini'` (built-in), `'external'` (pasted from other AI), `'manual'` (form entry). This is metadata for analytics, not displayed to users.
- Cloud Functions codebase uses `"gustify"` in firebase.json — do NOT use `"default"` (would overwrite Boletapp functions).
- Rate limiting is implicit via Firebase Auth — one user can't flood the CF. Consider adding explicit per-user rate limits in the CF if needed.
- The `functions/` directory has its own tsconfig (CommonJS). The Zod schema needs to be duplicated or shared. Simplest approach: define the schema in the CF and import the type. The client has its own copy in `src/types/recipe.ts`.
