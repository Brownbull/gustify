# Story 2.3: Allergen Filtering in Recipe Discovery

## Intent
**Epic Handle:** "No unsafe recipe reaches the screen"
**Story Handle:** "This story activates the bouncer -- no unsafe recipe gets past the door."

## Status: drafted

## User Story
As a user with dietary restrictions, I want recipes containing my allergens to be excluded from all results, so that I only see safe options.

## Acceptance Criteria

### AC-1: Allergen Exclusion
- **Given** a user with "gluten-free" restriction and "shellfish" allergy
- **When** browsing the recipe pool
- **Then** no recipe containing gluten-tagged or shellfish-tagged ingredients appears

### AC-2: Strict Filtering
- **Given** a recipe with 10 ingredients where 1 contains a user's allergen
- **When** filtering is applied
- **Then** the entire recipe is excluded (not just the ingredient)

### AC-3: No Restrictions Shows All
- **Given** a user with no dietary restrictions or allergies
- **When** browsing recipes
- **Then** all recipes are shown (no filtering applied)

### AC-4: Real-Time Filter Updates
- **Given** a user viewing recipes
- **When** they update their dietary profile (Story 2.2)
- **Then** the recipe list updates immediately without page reload

### AC-5: Dietary Compatibility for Diet Preferences
- **Given** a user with "Vegetariano" diet preference
- **When** browsing recipes
- **Then** recipes tagged with meat-based cuisines or containing meat ingredients are excluded

## Architectural ACs

- **AC-ARCH-LOC-1:** Allergen filter logic at `src/lib/allergen-filter.ts` (pure function, testable)
- **AC-ARCH-PATTERN-1:** Hybrid filtering: client-side exclusion pass on already-loaded recipes (architecture decision)
- **AC-ARCH-PATTERN-2:** RecipeStore reads dietary profile from authStore (one-way coupling, documented)
- **AC-ARCH-NO-1:** Allergen filtering MUST NOT have false negatives — a recipe with allergens must NEVER appear for a restricted user
- **AC-ARCH-NO-2:** No `not-in` Firestore queries for allergens (limited to 10 values)

## Dev Notes

- **V5 principle: "Health constraints are walls, not suggestions."** The allergen filter must be strict. When in doubt, EXCLUDE the recipe.
- The hybrid filtering approach (architecture decision): all recipes are loaded via real-time subscription. Client-side filtering applies allergen exclusion. This avoids Firestore's `not-in` limit (10 values).
- Allergen filtering runs as the FIRST filter in the chain: `allRecipes -> allergenFilter -> dietFilter -> searchFilter -> complexityFilter -> sortByMatch`. Safety before convenience.
- For "Vegetariano" / "Vegano" filtering, the initial implementation can be simple: check ingredient categories from the canonical dictionary. If an ingredient is in the "Proteinas" category with a meat-related name, exclude for vegetarian. This is imperfect but safe as a starting point.
- Ingredients without a canonicalId (from Gemini-generated recipes with unmatched names) are NOT excluded. This is a deliberate trade-off: we don't want to hide recipes just because ingredient names don't match perfectly. The risk is that an allergen-containing ingredient with a non-standard name might slip through. Mitigation: improve canonicalId matching over time.
- The canonical ingredients are already loaded by the app (used for item mapping). Reuse that data for the allergen lookup map.
